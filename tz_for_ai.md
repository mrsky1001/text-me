# Техническое задание (ТЗ) для AI: Мессенджер с проверкой английского (MVP)

Это пошаговое руководство (Prompt-план) для AI-ассистента по поэтапной реализации проекта. Каждый чекбокс соответствует задаче, которую можно передать нейросети (мне) в работу напрямую.

## Этап 1: Инфраструктура и Бэкенд (Node.js + Fastify)

### 1.1. Инициализация проекта
- [ ] Инициализировать `package.json`.
- [ ] Установить базовые зависимости: `fastify`, `fastify-cors`, `dotenv`.
- [ ] Создать базовую архитектуру файлов:
  - `/src/server.js` (точка входа)
  - `/src/routes/` (роуты)
  - `/src/services/` (бизнес-логика)

### 1.2. Настройка Базы Данных (PostgreSQL)
- [ ] Установить и настроить драйвер `pg` (или легковесный ORM/Query Builder).
- [ ] Создать модуль подключения к БД (`/src/db/index.js`).
- [ ] Написать скрипты/миграции для создания таблиц (DML):
  - `users` (id, username)
  - `chats` (id, created_at)
  - `chat_members` (chat_id, user_id)
  - `messages` (id, chat_id, sender_id, text, created_at, is_corrected, correction_data)

### 1.3. Базовый REST API (Заглушки)
- [ ] Написать роут `POST /api/users` для входа (псевдо-авторизация по username).
- [ ] Написать роут `POST /api/chats` (создание чата).
- [ ] Написать роут `GET /api/chats/:chatId/messages` (получение истории переписки).

## Этап 2: Реалтайм (WebSocket / Socket.io)

### 2.1. Интеграция Socket.io
- [ ] Интегрировать `socket.io` поверх Fastify (`fastify-socket.io`).
- [ ] Настроить обработчик `connection`.
- [ ] Реализовать логику комнат (`rooms`): прослушивание ивента `join_chat` для подписки на чат.

### 2.2. Логика обмена сообщениями
- [ ] Обработчик события `send_message` (принимает `text`, `chat_id`, `local_id`).
- [ ] Сохранение сообщения в БД `messages`.
- [ ] Обратная отправка отправителю `msg_ack` с подтверждением записи (чтобы снять "часики" доставки).
- [ ] Рассылка в комнату `new_msg` для доставки сообщения получателю.

## Этап 3: Фронтенд Клиент (Vanilla JS)

### 3.1. Верстка (Telegram-style)
- [ ] Сверстать `index.html`: сайдбар (список чатов) и область диалога (сообщения + поле ввода).
- [ ] Написать `styles.css` с использованием CSS variables для будущей темной темы.
- [ ] Спозиционировать баблы сообщений (мои справа зеленые, чужие слева белые/серые).
- [ ] Подготовить `<template>` теги для динамической генерации сообщений в DOM.

### 3.2. Логика отправки без проверок
- [ ] Написать JS логику авторизации с сохранением в `localStorage`.
- [ ] Загрузка истории чата через REST API и отрисовка в DOM.
- [ ] Подключение клиента `socket.io` к серверу.
- [ ] Optimistic UI: при вводе текста добавлять сообщение в DOM моментально.
- [ ] Обработка входящих `new_msg` через сокеты и скролл вниз.

## Этап 4: Ядро проверки грамматики (LanguageTool API)

### 4.1. Архитектура сервиса проверки (с прицелом на LLM)
- [ ] Создать `src/services/checkerService.js`.
- [ ] Описать базовый класс-интерфейс `TextAnalyzer` с методом `analyze(text)`.
- [ ] Вернуть из него строго типизированный ответ формата `AnalysisResult`: `{ original, corrected, has_errors, errors: [] }`.

### 4.2. Интеграция LanguageTool 
- [ ] Реализовать класс `LanguageToolAnalyzer extends TextAnalyzer`.
- [ ] Написать `fetch` запрос к публичному API (или локальному инстансу) LanguageTool.
- [ ] Написать логику парсинга массива `matches` с конца в начало для формирования чистого `corrected` текста.

### 4.3. Связка с WebSocket
- [ ] Изменить хэндлер `send_message`: после сохранения оригинала добавить асинхронный вызов `LanguageToolAnalyzer`.
- [ ] После резолва проверки обновлять запись в БД (заполнять `correction_data`).
- [ ] Рассылать в комнату сокетов новый ивент `msg_checked` с результатами.

---

## Этап 5: Фронтенд — Отображение проверок и "Разбор"

### 5.1. Отрисовка исправлений (Sub-bubbles)
- [ ] В JS клиенте слушать WebSocket ивент `msg_checked`.
- [ ] Находить DOM-узел по `msg_id`.
- [ ] Если в результатах `has_errors === true`: рендерить пристыкованный снизу bubble с исправленным текстом и кнопкой (лампочкой 💡).
- [ ] Если `has_errors === false`: добавлять микро-индикатор `EN ✓` рядом с временем отправки оригинала.

### 5.2. Шторка (Bottom Sheet) "Разбор ошибок"
- [ ] Сверстать скрытую панель разбора ошибок. 
- [ ] Настроить JS-listener на клик по иконке 💡 в исправлениях.
- [ ] При клике парсить `correction_data` из DOM `dataset` и динамически собирать список ошибок в шторке (`Ошибка -> Правильно` + пояснение LT).
- [ ] Анимировать выезд шторки (CSS `transform: translateY`).

---

## Этап 6: Финализация и Edge Cases
- [ ] Проверка на русский язык (`detectedLanguage` от LT) — игнорировать ошибки, если текст русскоязычный.
- [ ] Graceful degradation: если LT API недоступен, сообщение не должно зависнуть (просто не появляется плашка).
- [ ] Полировка CSS анимаций и margin'ов баблов.
- [ ] Очистка инпутов и экранирование входящего текста от XSS.
