# Мессенджер для изучения английского языка (MVP)

## 🧠 Ключевые продуктовые решения (UX/UI & Flow)

1. **Когда происходит проверка: Строго ПОСЛЕ отправки (Асинхронно)**
   * Проверка *до* отправки ломает ядро мессенджера — скорость. Мессенджер должен летать. Пользователь нажимает «Отправить», сообщение мгновенно появляется в чате (Optimistic UI). Параллельно бэкенд шлет текст в LanguageTool. Как только приходит ответ — под сообщением плавно появляется плашка с исправлением.

2. **Как отображается исправление: Пристыкованный Sub-bubble**
   * Исправление выглядит как ответное сообщение (reply), но пришито снизу к оригинальному "пузырьку", без зазоров. 
   * Фон исправления бледнее (светло-серый), шрифт на 10-15% меньше основного.
   * Ошибки подсвечиваются мягким красным зачеркиванием, исправления — зеленым.

3. **Как работает кнопка "Разбор": Bottom Sheet (Шторка) / Modal**
   * Внутри плашки с исправлением есть иконка 💡 или текст `Разбор`.
   * По клику снизу выезжает компактная шторка: `[Твоя ошибка] -> [Правильно]` и ниже короткий текст от LanguageTool.

4. **Что делать со специфичным текстом:**
   * **Ошибок нет:** В строке с галочками прочтения добавляем маленькую зеленую галочку `EN ✓`.
   * **Текст на русском:** Если LanguageTool определяет `detectedLanguage` как RU — мы не выводим плашку с исправлениями. Чат работает как обычный.
   * **Смешанный текст:** В MVP отправляем фрагмент в LanguageTool с параметром `language="en-US"`. Если LT выдаст странный результат — это приемлемое ограничение MVP.

## 🏗 Проектирование системы

### 1-3. Описание, Цель и ЦА
* **Продукт:** Мессенджер, где люди общаются в реальном времени, а система мягко корректирует их английский на лету. 
* **Цель MVP:** Проверить гипотезу "обучение через общение", замерить retention и проверить асинхронное исправление.
* **ЦА:** A2-B1, знающие правила, но совершающие ошибки из-за языкового барьера.

### 4. User Flow (Пошагово)
1. Юзер заходит в приложение, выбирает чат с другом.
2. Печатает фразу с ошибкой: *"I wants to go in cinema"*.
3. Жмет `Send`. Сообщение мгновенно улетает в UI (зеленый бабл).
4. Через 0.5 - 1 сек (пока друг читает сообщение) под баблом плавно разворачивается область: *"I **want** to go **to the** cinema"*.
5. Юзер кликает на иконку 💡 на этой плашке. Выезжает шторка с объяснением.

### 6. Логика LanguageTool (LT)
* **Endpoint:** `POST https://api.languagetoolplus.com/v2/check` (или self-hosted).
* **Payload:** `text=I+wants+to+go&language=en-US`.
* **Обработка ответа:** Идем по массиву `matches` **с конца в начало** (чтобы `offset` не съезжали), подставляя замены из `replacements`.

### 7. Архитектура
**Backend (Node.js + Fastify):**
* `src/api/routes` — REST эндпоинты (авторизация, списки чатов).
* `src/ws/socketManager.js` — модуль работы с подключениями, комнаты (rooms).
* `src/services/checkerService.js` — изолированный сервис отправки текста в LT. 

**Frontend (Vanilla JS):**
* `api.js` — HTTP запросы.
* `ws.js` — управление WebSocket.
* `ChatUI.js` — логика отрисовки DOM.

### 8. База данных (PostgreSQL)
* **`users`**: `id`, `username`
* **`chats`**: `id`, `created_at`
* **`chat_members`**: `chat_id`, `user_id`
* **`messages`**: `id`, `chat_id`, `sender_id`, `text`, `created_at`, `is_corrected` (boolean), `correction_data` (JSONB) — содержит ответ от LT.

### 9-10. Реалтайм (WebSocket) & API
1. Client: `WS SEND { type: 'chat_msg', chat_id: 1, text: "hi", local_id: "temp-123" }`
2. Server: Сохраняет в БД `messages`. Подтверждает (Ack) `local_id`.
3. Server: `WS SEND to Receiver { type: 'new_msg', id: "uuid", text: "hi" }`.
4. Server: Запрашивает LT в фоне. Обновляет БД.
5. Server: `WS SEND to Room { type: 'msg_checked', msg_id: "uuid", correction: {...} }`.
6. Client: Находит в DOM элемент с `msg_id` и дорисовывает sub-bubble.

### 11. Обработка ошибок
* **API LanguageTool недоступен:** Ставим метку без проверки, не прерываем UI.
* **Долгий ответ LT:** Не проблема (асинхронно). Плашка может появиться и через 3 секунды.

### 13. Подготовка к переходу на AI (LLM)
* Создаем базовый интерфейс `TextAnalyzer`. Метод: `async analyze(text: string): Promise<AnalysisResult>`.
* `AnalysisResult` строго типизирован: `original`, `corrected`, `has_errors`, `explanations: [...]`.
* В будущем просто пишем `LLMAnalyzer extends TextAnalyzer` и подменяем класс.

### 14. План разработки
1. **Этап 1 (Чат):** Верстка, Node.js, PostgreSQL с базовым CRUD и WS обменом сообщениями.
2. **Этап 2 (Проверка):** Интеграция LT API на бэкенде.
3. **Этап 3 (UI):** Обработка WS ивента `msg_checked` на клиенте, отрисовка плашек исправления, верстка шторки "Разбор".

### 16. Рекомендуемый Стек для MVP
* **Backend:** Node.js + Fastify. `socket.io` (для простоты комнат).
* **Database:** PostgreSQL (ради хранения JSONB для `correction_data`).
* **Frontend:** Vanilla JS (ES Modules) + CSS Variables (для темной/светлой темы). DOM обновляется точечно.
* **Деплой:** Railway.app или Render.com. Простота деплоя Node + PostgreSQL в пару кликов.
